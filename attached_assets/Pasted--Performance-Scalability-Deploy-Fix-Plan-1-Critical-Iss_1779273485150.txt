# Performance + Scalability + Deploy Fix Plan

## 1. 🔴 Critical Issues
- Fix Vercel deploy blocker in `server.ts`: `/api/teacher/dashboard` uses `teacherDashboardCacheKey` without declaring it. Add the cache key + cached read before dashboard DB work, then keep the existing `setCachedApiResponse(...)`.
- Remove wrong dashboard cache usage from `/api/teacher/progress` and `/api/teacher/results`; both currently use `teacher-dashboard:${requestedUserId}` and can return the wrong response shape.
- Replace global `app.use('/api/', apiLimiter)` for realtime routes. Current limit is `200 / 15 min`; student polling every 3s reaches ~300 requests/15min, so live quiz users get rate-limited mid-session.
- Migrate live quiz state out of in-memory `Map`/`Set` (`rqSessions`, `rqPins`, `rqCompletedSessions`, `rqAnswerProcessing`). This cannot scale horizontally and breaks across serverless instances.
- Stop using `fetchAllAttemptRows()` for dashboards/reports. It does `quiz_attempts.select("*")` or `attempts.select("*")` and filters in Node, which becomes a full-table bottleneck.
- Rotate exposed secrets immediately: Supabase service role key, OpenAI API key, Telegram bot token, and any Vercel env copies. Treat the pasted keys as compromised.

## 2. 🟠 Performance Issues
- Live quiz answer handling recalculates and broadcasts the full leaderboard on every answer; at 1k-10k users this becomes CPU-heavy and payload-heavy.
- `rqPersistSession(session)` writes the whole live quiz session JSON to Supabase after joins/answers/question changes; this creates write amplification.
- Chat realtime has profile N+1 queries on every inserted message in `LiveSessionJoin.tsx` and `LiveSessionRoom.tsx`.
- `/api/teacher/dashboard`, `/api/teacher/progress`, `/api/teacher/results`, `/api/admin/analytics`, and admin reports do large in-memory aggregation instead of SQL aggregation.
- `session_chat_messages` GET returns the whole session chat ordered ascending with no cursor pagination.
- Many endpoints still use `select("*")`, including quizzes, profiles, classes, questions, attempts, and certificates.
- `npm run lint` currently fails because `tsconfig.json` includes `artifacts/mockup-sandbox` plus real product TS errors. Add product-only TypeScript config before relying on CI.

## 3. 🟡 Optimization Opportunities
- Add missing hot-path indexes beyond `008_performance_indexes.sql`: `quiz_attempts(quiz_id, student_id, completed_at)`, `quiz_attempts(student_id, completed_at)`, `session_chat_messages(session_id, created_at)`, `live_sessions(status, scheduled_at)`, `notifications(user_id, created_at)`, `classes(teacher_id)`, `quizzes(course_id, status)`.
- Replace broad API payloads with explicit select lists.
- Add cursor pagination to reports/results/chat: `limit`, `cursor`, `from`, `to`, `quizId`, `studentId`.
- Add Redis-backed cache for dashboards and auth/profile lookups; current in-memory cache only works per instance.
- Frontend should reduce fallback polling to reconnect-only or 15-30s visibility-aware polling; use realtime events as primary source.
- Virtualize large tables in teacher/admin reports and reduce chart data to pre-aggregated daily buckets.

## 4. 🧠 Architecture Redesign
- Keep the API stateless. Use Postgres/Supabase as source of truth, Upstash Redis for cache/rate limits/live state, and a separate realtime gateway for live quiz.
- For 10k+ live quiz users, run a dedicated WebSocket service on Fly/Railway/Render. Vercel serverless should not own WebSocket session state.
- Live quiz flow: start session writes metadata to SQL and Redis; PIN maps to session in Redis with TTL; joins go to Redis sets/hashes; answers use idempotency keys; leaderboard uses Redis sorted sets; broadcasts send deltas/top-N only; final report persists to SQL.
- Chat flow: insert message with denormalized `sender_display_name` and `sender_avatar_url`, broadcast that row, and fetch history with cursor pagination.
- Heavy jobs/reports: use Vercel Cron + Upstash QStash for current Vercel setup. Move to BullMQ only if a persistent Node worker is deployed.

## 5. 🛠 Step-By-Step Action Plan: 7 Days
- Day 1: Fix `teacherDashboardCacheKey`, remove wrong cache keys from progress/results, rotate secrets, add `tsconfig.prod.json` excluding `artifacts`, and make CI run product TypeScript checks.
- Day 2: Split rate limiters by route: auth strict, normal API moderate, realtime answer/chat separate, polling fallback lenient or disabled. Use Redis store for rate limits.
- Day 3: Replace `fetchAllAttemptRows()` usage in teacher dashboard/results/progress with filtered DB queries and SQL aggregates. Remove `select("*")` from those endpoints.
- Day 4: Add missing SQL indexes and verify with Supabase `EXPLAIN` on dashboard, reports, live sessions, chat, and attempts queries.
- Day 5: Move live quiz state to Redis: sessions, PINs, participants, answer idempotency, and leaderboard sorted sets. Stop persisting full session JSON on every answer.
- Day 6: Fix chat/profile N+1, add chat cursor pagination, reduce frontend polling, and add virtualization for large report/result lists.
- Day 7: Load test with k6: normal API, teacher dashboard, live quiz 1k/5k simulated users, chat bursts, and reports. Add P95 latency/error-rate alerts.

## 6. ⚡ Final Verdict
- Current capacity estimate: normal usage may survive a few hundred to ~1k light users, but live quiz/chat will struggle much earlier because of polling, memory state, full leaderboard broadcasts, and full-table attempt reads.
- Current live quiz estimate: risky above ~100-300 active realtime students per session; 10k is not realistic with the current architecture.
- After Day 1-3 fixes: deploy becomes stable, dashboard/report latency improves, and 1k light users becomes more realistic.
- After full Redis + SQL aggregation + realtime redesign: 10k+ concurrent users is achievable, with normal API P95 near or under 200ms for cached/simple endpoints and heavy reports handled asynchronously.
